Playbooks

End-to-end pipeline architectures for four high-priority insurance document intelligence use cases — stage by stage, with technology stack, data posture constraints, and key decision points.

Mailroom Straight-Through Processing
Automated intake, classification, and routing of inbound insurance documents — from raw email or portal upload to indexed, typed, and queued document record. Target: 85–90% STP rate with a well-designed exception path for the remainder.
Posture A (commercial) · Posture B (claims intake) No LLM required Target STP: 85–90%
1
Document Ingestion & Format Normalization
Receive documents from all inbound channels — email attachments, portal uploads, fax-to-PDF, API submissions. Normalize all formats to a canonical PDF representation. Split multi-document packages into individual files.
Email parser (IMAP / Exchange / O365) PyMuPDF (format normalization) pdfplumber (page splitting) TIFF → PDF converter
2
Quality Assessment & OCR Gate
Score each page for scan quality (DPI, blur, skew, contrast). Digital-native PDFs with AcroForm fields bypass OCR entirely. Scanned or image-only PDFs route to the appropriate OCR tier based on quality score.
OpenCV (DPI, blur, skew detection) pdfplumber (AcroForm field check) Tesseract 5 (clean scans ≥ 300 DPI) PaddleOCR v4 (complex layouts, low-res)
Quality gate threshold: <150 DPI or blur score >0.7 → flag for rescan request before processing.
3
Document Type Classification
Identify the primary document type. Rules-based first — ACORD form numbers appear in headers and are deterministic. Novel or ambiguous formats escalate to LLM classification as a fallback only.
Compiled regex (ACORD form numbers) Header keyword signatures Page count + layout heuristics LLM escalation (ambiguous only)
Known types: ACORD 125/126/130/140/160, loss run, SOV, policy, endorsement, IA report, broker cover letter, claim file.
4
Subtype Classification
Resolve the document subtype within the identified type — ACORD form variant, line of business, transaction type (new business vs. renewal vs. endorsement), and carrier-specific variants.
ACORD form variant rules (125 GL / WC / Umbrella) Line of business keyword rules Transaction type classifier (new / renewal / endorsement)
5
Metadata Extraction (Cat 1)
Extract key indexing fields from the classified document. These are deterministic structured fields — no LLM required.
Named insured Policy number FEIN / NPI Effective / expiry dates Broker code Claim number (claims intake)
Compiled regex (policy #, FEIN, date patterns) pdfplumber (digital AcroForm fields)
6
Completeness Validation
Check that all required metadata fields are present and in valid format for the classified document type. Incomplete records route to an exception queue — not to the downstream pipeline.
Completeness schema per document type Field format validators Exception queue router
7
Index & Route
Write the classified, validated document record to the document index. Trigger routing rules to assign to the correct workflow queue — UW intake, claims assignment, renewals, endorsement processing.
Document index (MongoDB / Elasticsearch) Routing rules engine Queue assignment (UW / Claims / Renewals / Endorsements) SLA timer start

Exception Path (10–15%)

Low-confidence classification, missing required metadata, and quality-failed documents route to a human review queue. The exception queue design is as important as the STP pipeline — poorly designed exception handling becomes the operational bottleneck.

Posture Split

Commercial submissions are Posture A. Claims intake documents (which may include medical attachments, police reports) should be treated as Posture B. Build the pipeline with posture tagging per document type from step 3 onward.

Single ACORD / EMOD Extraction
Deterministic structured extraction from ACORD commercial application forms (125, 126, 130, 140) and workers' compensation EMOD worksheets. Fully LLM-free — Cat 1 and Cat 5 only. Output is a structured JSON record ready for the submission or rating system.
Posture A No LLM — fully deterministic
1
Document Condition Detection
Determine whether the ACORD form is a digital PDF with AcroForm fields or a scanned/image-only PDF. This determines which extraction path to follow — they have very different accuracy profiles and tool requirements.
pdfplumber (AcroForm field detection) PyMuPDF (PDF metadata + page analysis)
2
Path-Dependent Extraction
Two parallel paths based on document condition. Digital path achieves near-100% accuracy. Scan path accuracy depends on scan quality and form complexity.
Path A — Digital PDF (AcroForm fields present)
Stack
pdfplumber (direct field read) PyMuPDF (AcroForm values)
Accuracy
Near-100% — no OCR involved
Path B — Scanned / Image PDF
Stack
Tesseract 5 / PaddleOCR v4 Form structure detection (Cat 5) Field position mapping
Accuracy
90–96% (clean scan) · 75–85% (degraded)
3
Structured Field Extraction (Cat 1)
Extract all deterministic structured fields from the ACORD form. Each field type has a compiled regex pattern and value normalizer.
Named insured + DBA FEIN SIC code Mailing + location address Effective / expiry dates Coverage limits Deductibles Total premium Years in business Annual revenue Employee count
Compiled regex per field type Value normalizers (date, currency, address)
4
Categorical Extraction (Cat 2)
Resolve ontology-anchored categoricals from the extracted SIC code and business description. No LLM — crosswalk tables and ontology index handle this deterministically.
SIC → NAICS crosswalk Industry vertical mapping Business description keyword classifier Class of business ontology index
5
EMOD Table Parsing (WC forms only)
Parse the experience modification worksheet when present. The EMOD table has a predictable column structure — prior policy years, primary losses, excess losses, expected losses, and the computed mod factor. Validate the mod factor is consistent with the actuarial formula.
Cat 5 table parser Prior year columns (3–5 years) Primary loss extraction Excess loss extraction Expected loss extraction Mod factor (computed + stated)
Mod factor validation: flag if extracted factor is outside 0.50–2.00 range or inconsistent with loss inputs.
6
Validation
Apply field-level and cross-field validation rules. Failures route to a human correction queue — do not pass invalid records downstream.
Required field completeness check Date logic (effective < expiry) Deductible < limit check FEIN format validator (XX-XXXXXXX) SIC code valid range check EMOD range check (0.50–2.00)
7
Structured Output
Emit a validated JSON record conforming to the submission intake schema. The record includes all extracted fields, extraction confidence per field, the source path (digital vs. scan), and EMOD fields if applicable.
JSON schema validator Per-field confidence scores Extraction path tag (digital / scan) API output → submission system / GKR pipeline

Digital vs. Scan Accuracy Gap

Digital AcroForm PDFs produce near-100% extraction accuracy. Scanned forms drop to 90–96% on clean scans and 75–85% on degraded input. Carriers should report submission quality rates to brokers and incentivize digital submission formats.

EMOD Source of Truth

Always extract both the stated mod factor and the calculated mod factor from the worksheet inputs. Discrepancies between the two are a data quality flag that should route to the underwriter — not be silently resolved by the pipeline.

Context Extraction for Conversational GenAI — Claims
Prepare heterogeneous claims file documents (FNOL, police reports, IA reports, medical records, adjuster notes) as structured, redacted, retrievable context for a conversational AI interface. Enables adjusters to ask natural language questions about a claim grounded in the actual file.
Posture B — PHI present T4 Local LLM (VPC)
1
Redaction — Must Be First
Strip all PII and PHI before any other processing or LLM contact. This is not optional — claims files contain claimant names, SSN, medical record numbers, DOB, and provider identifiers. Redaction produces an audit log of what was removed and why, required for HIPAA compliance documentation.
MUST run before all other stages Compiled regex (SSN, TIN, DOB, phone, email, MRN) spaCy NER (PERSON, ORG, GPE) Medical record number patterns Redaction audit log (HIPAA evidence) Determinism validator (same input → same output)
2
Document Type Classification
A single claims file package may contain 8–15 distinct document types. Classify each document separately — each type requires a different extraction schema in the next stage.
FNOL form classifier Police report classifier IA report classifier Medical record classifier Adjuster note classifier Coverage letter classifier
3
Type-Aware Extraction
Each document type runs through its own extraction schema. This is the critical difference from generic RAG — insurance-specific extraction produces structured fields alongside narrative, enabling both precise lookup and semantic retrieval.
FNOL → Cat 1 + Cat 3
Date of loss Claim number Policy number Reported injuries (Cat 1) Loss narrative (Cat 3 verbatim)
Police Report → Cat 3
Verbatim narrative capture Violation / charge codes (Cat 2)
IA Report → Cat 3 + Cat 4
Damage description (Cat 3) Causation assessment (Cat 4) Repair scope interpretation (Cat 4)
Medical Records (post-redaction) → Cat 1
ICD-10 diagnosis codes CPT treatment codes Treatment dates Provider specialty
4
Insurance-Specific Chunking
Segment documents by logical section and event timeline — not arbitrary token windows. A police report sections differently from an IA report. Event-aligned chunks enable timeline-based retrieval: "what happened on date X?" retrieves the right chunk regardless of which document it came from.
Document-type-aware section boundary detection Event timeline parser (dates, sequences) Cross-document event alignment Chunk metadata tagging (doc type, section, date)
Do not use generic text splitters (LangChain RecursiveCharacterTextSplitter etc.) — they cut across section boundaries and produce incoherent chunks for insurance content.
5
Claim Context Assembly
Combine all extracted structured fields and narrative chunks into a single per-claim context object. This is the queryable representation of the entire claim file.
Structured field store (per claim) Narrative chunk index (per claim) Cross-document event timeline Document inventory manifest
6
Retrieval Index (Per Claim)
Build a searchable index scoped to the claim. Local vector store or structured search — Posture B prohibits cloud vector databases (Pinecone, Weaviate cloud, etc.).
Local vector store (Qdrant local / Chroma) Structured field search layer Per-claim namespace isolation
7
Query-Time Retrieval + Generation
Adjuster asks a natural language question. The retrieval layer fetches relevant structured fields and narrative chunks. The T4 local LLM generates a grounded answer with source attribution — identifying which document and section each fact came from.
Ollama + Llama 3.3 70B (local, Posture B) Retrieval layer (structured + semantic) Governed prompt template Source attribution (doc type + section) Response formatter
The LLM retrieves and synthesizes — it does not generate coverage positions. Coverage determination stays in the rules layer.

Redaction Must Be First — No Exceptions

The LLM must never see un-redacted claimant PII or PHI. Redaction is the first stage in the pipeline, not an optional pre-processing step. Any architecture that passes raw claims documents to an LLM before redaction is non-compliant with HIPAA and most carrier data governance policies.

No Generic Chunking

Arbitrary token-window chunking (500 tokens, 20% overlap) cuts across section boundaries in police reports, IA reports, and medical records, producing incoherent retrieval. Insurance-specific chunking by document section and event timeline is required for reliable retrieval quality.

Coverage Determination Boundary

The conversational AI answers factual questions about the claim file. It does not determine coverage, set reserves, or make liability decisions. These remain with the adjuster. The system prompt must explicitly prohibit coverage assertions.

Local Infrastructure Requirement

Posture B requires Llama 3.3 70B via Ollama within the carrier VPC. GPU requirement: 2× A100 80GB minimum. Local vector store (Qdrant or Chroma) replaces cloud vector database services. Plan infrastructure procurement timeline before committing to delivery date.

UW Documentation Context Extraction + Summarization
Extract and structure all documents in a commercial submission package — ACORD application, loss run, SOV, inspection report, broker cover letter — then run GKR eligibility gates and LLM exposure interrogation to produce a governed, chunk-traced underwriting summary.
Posture A (commercial submissions) Non-Chinese LLM (cloud)
1
Submission Package Classification
Identify each document type in the submission package. A typical commercial submission contains 4–8 distinct document types, each requiring a different extraction schema.
ACORD application classifier (125 / 126 / 130 / 140) Loss run identifier (column structure, date pattern) SOV identifier (COPE field presence) Inspection report classifier Broker cover letter classifier
2
Type-Aware Extraction
Each document type runs its own extraction schema — same principle as claims but optimized for underwriting content types.
ACORD Application → Cat 1 + Cat 5
Named insured, FEIN, SIC Operations description (Cat 3) Locations, limits, deductibles Form structure detection (Cat 5)
Loss Run → Cat 1 + Cat 5 (table)
Claim dates, amounts, types Open vs. closed status Large loss identification (>$100K) Table structure detection (Cat 5)
SOV → Cat 1 COPE + Cat 5
Construction type Occupancy classification Protection class Building value / TIV Year built
Inspection Report → Cat 3 + Cat 4
Condition narrative (Cat 3) Deficiency findings (Cat 4) Recommendation interpretation (Cat 4)
3
GKR Three-Gate Eligibility — Runs Before LLM
The deterministic eligibility pipeline runs on the extracted structured fields before any LLM contact. Gate 1 short-circuits on hard exclusions. Gates 2 and 3 resolve appetite and operations. The LLM only sees submissions that have passed or conditionally passed the gates.
Gate 1 — Hard Exclusions (SIC → exclusion matrix) Gate 2 — Vertical Appetite (SIC → appetite chunks) Gate 3 — Operations Exclusion (keyword match on description)
Gate 1 DECLINE = pipeline terminates. No LLM cost incurred on ineligible submissions. Eligibility verdict + confidence written to decision trace.
4
Context Assembly
Combine gate outputs, structured fields, and narrative chunks into a governed context window for the LLM. The context is curated — not the raw documents. This limits token cost, reduces hallucination surface, and ensures the LLM operates on verified extracted data.
Gate verdict + confidence scores Structured fields (insured, SIC, operations, limits) Loss run summary (frequency, severity, large losses) SOV summary (TIV, construction mix, protection class) Inspection findings narrative
5
UW-CP-001 Exposure Interrogation
The LLM analyzes the assembled context against the 9-pattern UW-CP-001 cognitive pattern. Each pattern probes a distinct exposure dimension. The LLM's scope is constrained by the governed prompt template — it cannot go beyond the 9 patterns or assert facts not in the context.
Claude / GPT-4 (governed prompt) Premises & operations liability Products & completed operations Contractual liability Professional / E&O exposure Property concentration Catastrophe exposure Loss history analysis Management / governance risk Regulatory / compliance exposure
6
Summary Generation
Generate a structured underwriting summary: eligibility verdict, top exposure flags, loss history narrative, open questions for the underwriter. Every assertion in the summary traces to a chunk ID or gate output — no unsupported claims.
Eligibility verdict + confidence Top 3 exposure flags (with chunk citations) Loss history narrative Open questions for underwriter Recommended next steps
7
GKR Format Output
Emit the final response in GKR format — the standard agent response schema used across the platform. The decision trace makes every gate result, LLM input, chunk citation, and confidence score auditable.
response — flat markdown narrative decision_trace — gate results, verdict, confidence chunk_ids — cited knowledge sources version — cognitive pattern version (UW-CP-001 v2.0)

GKR Gates Must Run First

The three-gate deterministic eligibility pipeline always runs before the LLM. This eliminates LLM cost on clearly ineligible submissions (Gate 1 DECLINE short-circuits) and ensures the LLM never produces a positive exposure narrative for an ineligible risk.

Context Not Raw Documents

The LLM receives curated structured context — not raw document content. Passing raw ACORD PDFs to an LLM is expensive, slow, and produces lower-quality outputs than passing well-structured extracted fields. Extract first, then govern the context window.

GKR Format Required

All outputs must conform to GKR format: a flat markdown response and a structured decision_trace. Free-form LLM output without a decision trace is not auditable and cannot be used for underwriting decisions. This is a hard requirement, not a preference.

Non-Chinese LLM Policy

Commercial submissions contain financially sensitive business information (revenue, employee count, operations detail). Apply a non-Chinese LLM policy as a default — use Anthropic Claude or OpenAI GPT-4. The Insights site decision matrix provides the full provider reference.
