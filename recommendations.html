Solution Recommendations

Buy, Build, or Blend — prescriptive recommendations for four high-priority insurance document intelligence use cases, with accuracy benchmarks, product landscape assessment, and risk summary.

Evaluation Framework

Accuracy Fit

Does the solution match the extraction complexity and document type? Can it meet production accuracy thresholds on the carrier's actual document mix?

Posture Compliance

Can the solution operate within the carrier's data sovereignty constraints? Posture B (no PHI to cloud) disqualifies most vendor LLM and vector database options.

Control & Auditability

Can every output be traced, audited, and governed? Eligibility decisions and coverage determinations require a full decision trace — not just an answer.

Recommendation — Build

Digital ACORD: pure Build — pdfplumber reads AcroForm fields at 98–99.5% accuracy with zero marginal cost. No vendor is justifiable. Scanned ACORD: selective Buy — Azure Form Recognizer for the scan path, or Docling (open source) if Posture B is required. EMOD: Build — deterministic table parsing, no LLM. The business case for buying IDP tooling on this use case is weak: the accuracy gap over open source on digital forms is negligible, while the cost gap is significant at volume.

Recommendation — Build

Posture B eliminates vendor LLM and cloud vector database options — this use case must be built. Redaction (spaCy + regex) is non-negotiable as the first pipeline stage. Type-aware extraction must be built with insurance-specific schemas per document type — generic IDP tooling produces inadequate output for claims content. Chunking must be insurance-specific — generic text splitters cut across section boundaries and degrade retrieval quality. LLM generation: Llama 3.3 70B via Ollama within carrier VPC. Only viable buy candidate: Azure Form Recognizer for handwritten content on specific document sub-types. Infrastructure requirement: 2× A100 80GB GPU for the local LLM.

Recommendation — Blend

Buy the extraction layer — Azure Document Intelligence or an IDP provider (Reducto/Hyperscience) for multi-document submission packages. Build the eligibility rules engine — this is carrier-specific, deterministic, and must never be outsourced; it is the trust anchor of every underwriting decision. Buy Claude or GPT-4 (via Azure OpenAI or Anthropic API) for exposure interrogation and summarization — Posture A allows cloud LLM and frontier models significantly outperform local alternatives for nuanced exposure analysis. Build the output governance layer — structured decision trace, chunk citations, confidence scoring. Every assertion in the underwriting summary must trace to a governed source.

Data Posture — Per Use Case

Use Case 01
Mailroom STP — Posture A (commercial) · Posture B (claims intake)
Split Posture
Document StreamPostureImplication
Commercial submissions (ACORD apps, loss runs, SOVs) A — Cloud-safe Azure Form Recognizer, cloud classification APIs, and managed OCR services are permissible. No PHI present in standard commercial submissions.
Claims intake (FNOL, IA reports, medical attachments, police reports) B — No PHI to cloud PHI present from intake. Cloud OCR and classification APIs prohibited for this stream. Tesseract/PaddleOCR (T1) must handle the scan path. Rules-based classifier only — no LLM escalation via cloud API.
Mixed packages (submission with medical loss history attached) B — treat as B When posture is ambiguous, default to B. Build posture tagging into the classification step so downstream stages inherit the correct constraint automatically.

Key design requirement

The mailroom pipeline must tag each document with its posture at classification time. Every downstream stage (OCR vendor selection, metadata extraction method, routing destination) must inherit and enforce that posture tag. A single mis-tagged PHI document processed through a Posture A path is a compliance incident.

What Posture B prohibits

Cloud LLM APIs (OpenAI, Anthropic, Azure OpenAI, Google Vertex) · Cloud vector databases (Pinecone, Weaviate Cloud, Azure AI Search) · Any managed service where document content leaves the carrier network perimeter.

What Posture B permits

Ollama + Llama 3.3 70B (local, carrier VPC) · Qdrant or Chroma (local vector store) · Tesseract 5 / PaddleOCR (local OCR) · spaCy NER (local) · pdfplumber (local).

Infrastructure implication: Posture B requires GPU infrastructure within the carrier VPC for the local LLM. Minimum: 2× NVIDIA A100 80GB (or equivalent) for Llama 3.3 70B inference. This is a procurement and lead-time consideration — plan 8–12 weeks for GPU acquisition and VPC deployment before committing to a delivery date.

Practical recommendation

Commercial GL, Property, and E&O submissions are cleanly Posture A — enable cloud LLM (Claude, GPT-4) for the summarization layer without restriction. WC and auto BI submissions should be evaluated per account: if the loss run contains claimant-level medical detail, route that submission through the Posture B pipeline (local LLM, no cloud vendor contact). Build the routing decision into the classification step rather than making it a manual underwriter judgment.

Risk Summary

High

Posture Creep

Vendors default to Posture A. A single PHI document in a submission package can breach compliance if cloud processing is enabled without posture tagging. Build posture classification into the pipeline from day one.

High

Hallucination Without Governance

Ungoverned LLM outputs have 15–25% hallucination rates on insurance content. Without governed prompts, grounding context, and decision traces, LLM outputs are not auditable for underwriting or claims decisions.

High

Vendor Lock-in (IDP / Insurtech)

IDP and insurtech vendors train on your document corpus. Switching cost after 12+ months of production use is significant. Define exit criteria and data portability requirements before signing.

Medium

Accuracy Cliff on Degraded Scans

Extraction accuracy drops sharply below 300 DPI. Fax-to-PDF intake routinely produces sub-150 DPI documents. Carrier must publish minimum quality standards to brokers and build a rescan request workflow.

Medium

Per-Page Cost at Scale

Hyperscaler and IDP per-page pricing becomes expensive above ~50,000 pages/month. Fixed infrastructure cost of a self-hosted build is lower above that threshold. Run TCO analysis before committing to managed services at volume.

Medium

LLM Provider Geopolitical Risk

Models from Chinese-origin companies (Qwen, DeepSeek, ERNIE) are subject to China's National Intelligence Law. Routing sensitive commercial or claims data through these models creates geopolitical and governance risk. Apply a non-Chinese LLM policy as a default.

Benchmark Methodology

The accuracy ranges in this document are indicative benchmarks — informed directional estimates derived from four sources, not figures from a controlled empirical study on carrier-specific documents.

Sources used

  • Published technology benchmarks — Tesseract 5, PaddleOCR v4, Azure Form Recognizer, AWS Textract vendor documentation
  • Academic research — i2b2 medical NER challenge (PHI redaction recall), TruthfulQA and FactScore studies (LLM hallucination rates), RAG evaluation literature (retrieval precision)
  • IDP industry reports and vendor case studies for insurance mailroom STP rates
  • First-principles reasoning for deterministic systems (e.g. AcroForm field extraction, hard exclusion rule matching)

Known limitations

  • Figures are not validated against this carrier's specific document corpus, form variants, or broker submission mix
  • OCR accuracy is highly sensitive to scan resolution and document condition — ranges widen significantly on degraded input
  • LLM hallucination rates vary by model version, prompt design, and domain — ranges reflect published averages, not production measurements
  • STP rates are informed by IDP vendor case studies which may reflect optimised deployments

Recommended next step: Validate against a stratified sample of 200–300 carrier documents (clean digital, clean scan, moderate scan, degraded scan) using a structured evaluation harness before committing to production accuracy thresholds or SLAs. These benchmarks provide the planning baseline — corpus evaluation provides the production baseline.
